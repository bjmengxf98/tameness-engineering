# Tameness Engineering（驯服工程）

> **智能体不是写出来的，是驯出来的。**

## 什么是 Tameness Engineering？

**Tameness Engineering，驯服工程**，是我提出的一个面向 AI 智能体开发的新概念。

它强调：大模型智能体的可靠性，不是靠一次性写好 prompt、加好约束、接好工具就能完成的，而是在真实环境中通过持续观察、校正、强化和稳定，慢慢“驯”出来的。

提示词工程给模型命令。  
上下文工程给模型环境。  
Harness 工程给模型工具。  
对齐工程给模型边界。  
**驯服工程给模型习性。**

## 核心判断

大模型不是传统软件函数。

传统软件更像：

```text
输入 A -> 执行规则 B -> 输出 C
```

而大模型智能体更像：

```text
输入 A -> 结合上下文、语义联想、工具环境、系统提示、历史反馈和模型自身倾向 -> 生成概率性的行为
```

它不是不会做事。  
恰恰相反，它太会做事了。

问题在于，它的能力在一开始往往是“野生”的。

它会补全、会猜测、会顺着用户说、会在信息不足时给出看似合理的答案、会在长上下文中丢失目标、会在工具调用中出现过度或不足。

所以，智能体开发的核心不只是“控制模型”，而是把模型的野生能力驯化成稳定、可预测、可纠正、可协作的行为。

## 驯服工程的循环

```text
观察 -> 识别 -> 校正 -> 强化 -> 稳定 -> 重复
```

### 1. 观察

不要只看模型最后答得好不好。  
要看它是如何行动的。

它是在推理，还是在补全？  
它是在核实，还是在猜测？  
它是在执行任务，还是在讨好用户？  
它是在遵守流程，还是只是在表面模仿流程？

### 2. 识别

找到反复出现的跑偏模式：

- 信息不足时乱猜
- 工具调用过度或不足
- 长任务中目标漂移
- 对用户过度迎合
- 遇到边界不停止
- 失败后编造成功结果
- 该问不问，该查不查
- 该停不停，该拒不拒

### 3. 校正

不要猛改，不要乱堆规则。

像“治大国如烹小鲜”一样，小步调整：

- 调整提示词
- 调整上下文顺序
- 增加检查点
- 明确工具调用条件
- 加入失败恢复流程
- 让模型在关键步骤自检
- 让模型在不确定时先核实

### 4. 强化

当模型做对时，要把正确行为固化下来。

好的行为要变成默认路径。  
可靠的动作要变成习惯。  
有效的流程要沉淀成框架。

### 5. 稳定

目标不是一次成功，而是多次稳定成功。

驯服工程不是临时调 prompt。  
它是智能体开发中的长期行为塑形。

## 和其他工程概念的区别

| 工程类型 | 关注点 |
|---|---|
| Prompt Engineering / 提示词工程 | 怎么命令模型 |
| Context Engineering / 上下文工程 | 怎么给模型环境 |
| Harness Engineering / 运行框架工程 | 怎么给模型工具和外壳 |
| Alignment Engineering / 对齐工程 | 怎么给模型价值和边界 |
| **Tameness Engineering / 驯服工程** | 怎么塑造模型习性 |

## 一个简单比喻

**Harness Engineering 是给马上马具。**  
**Tameness Engineering 是让马懂得配合。**

Harness 关注外部结构。  
Tameness 关注行为习性。

两者都重要，但不是一回事。

## 为什么重要？

随着 AI 智能体可以调用工具、写代码、访问文件、操作浏览器、连接日历、收发邮件、调用 API，问题已经不再只是：

> 模型能不能回答？

而变成：

> 智能体能不能稳定行动？

Tameness Engineering 正是为了描述这个问题。

它不是要削弱模型，而是要让模型的能力变得可用。  
不是把模型关进笼子，而是让它形成正确的行为习性。  
不是让模型绝对听话，而是让它有分寸地协作。
